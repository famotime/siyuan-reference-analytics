# 统计卡片规则与定义说明

本文基于当前代码实现整理“脉络镜（Context lens）”顶部统计卡片及其详情联动的统计规则、定义边界与使用场景。结论以当前仓库实现为准，不按 PRD 理想态做推断。

## 1. 分析前提

### 1.1 数据来源

当前分析快照由 `loadAnalyticsSnapshot()` 生成，引用数据来自两层：

1. `refs` 表中 `type = 'ref_id'` 的正式引用记录。
2. markdown fallback 解析出的内部链接引用。

markdown fallback 当前支持：

- `siyuan://blocks/<id>`
- `((block-id "title"))`
- `((block-id 'title'))`

对应实现：

- `src/analytics/siyuan-data.ts`
- `src/analytics/internal-links.ts`

注意点：

- 同文档自引用会被排除。
- fallback 引用会与 `refs` 结果合并，并按 `sourceDocumentId + sourceBlockId + targetDocumentId + targetBlockId` 去重。

### 1.2 分析对象

- 节点是“文档”。
- 边是“文档到文档的聚合引用关系”。
- 连通性、社区、桥接、路径、传播节点分析阶段都按无向图处理。

这意味着：

- 一条 `A -> B` 会让 `A` 与 `B` 在图里连通。
- 多个块从同一文档指向同一目标文档时，很多统计口径会按“文档对”去重，而不是按块级引用次数累计。

### 1.3 筛选与时间窗口

所有卡片都先经过统一样本筛选，再进入具体统计。

统一筛选入口：

- `filterDocumentsByTimeRange()` in `src/analytics/analysis.ts`

先应用的筛选条件：

- 笔记本 `notebook`
- 标签 `tag`
- 关键词 `keyword`
- 主题 `themeNames`

其中主题筛选并不是按“主题文档关系”过滤，而是检查文档的 `title + hpath + path + tags` 是否包含所选主题名。

时间窗口规则：

- `all`：直接保留筛选后的全部文档。
- `3d / 7d / 30d / 60d / 90d`：
  - 保留当前窗口内存在有效引用连接的文档；
  - 同时保留“当前窗口内最近创建或更新过、即使没有连接”的文档。

这也是为什么“最近编辑但暂时没连出去的文档”仍会出现在文档样本、孤立文档等卡片中。

### 1.4 当前实现中的“去重”原则

这是理解卡片值的关键。

- `summary.totalReferences` 不是原始引用行数，而是当前窗口内“唯一有向文档边”数量。
- 核心文档入链数按“不同来源文档数”统计，不按块级重复引用累加。
- 活跃关系详情里的入链/出链，也是按“不同对端文档”计数。
- 趋势卡片中的 `current.referenceCount / previous.referenceCount` 也是唯一有向文档边数量。
- 趋势里的“新增连接 / 断裂连接”按无向文档对比较。

## 2. 顶部统计卡片总览

卡片定义集中在：

- `buildSummaryCards()` in `src/analytics/summary-details.ts`

详情联动集中在：

- `buildSummaryDetailSections()` in `src/analytics/summary-details.ts`
- `src/App.vue`
- `src/composables/use-analytics.ts`

当前卡片如下：

| 卡片 | 值来源 | 核心口径 |
| --- | --- | --- |
| 文档样本 | `filteredDocuments.length` | 命中筛选条件且落入当前时间窗口样本的文档数 |
| 活跃关系 | `report.summary.totalReferences` | 当前窗口内唯一有向文档边数量 |
| 核心文档 | `report.ranking.length` | 当前窗口内至少有 1 个有效入链的文档数 |
| 趋势观察 | `risingDocuments.length + fallingDocuments.length` | 当前窗口与前一窗口相比发生变化的文档数 |
| 主题社区 | `report.summary.communityCount` | 去掉桥接节点后形成的多文档连通簇数量 |
| 孤立文档 | `report.summary.orphanCount` | 当前窗口内没有任何文档级连接的文档数 |
| 沉没文档 | `report.summary.dormantCount` | 孤立文档中，未活跃时长达到阈值的文档数 |
| 桥接节点 | `report.bridgeDocuments.length` | 断开后会显著削弱连通性的关键文档数 |
| 传播节点 | `report.summary.propagationCount` | 高频出现在关键最短路径上的文档数 |

显示控制规则：

- `showSummaryCards` 决定是否展示顶部卡片区。
- `showRanking` 控制“核心文档”。
- `showTrends` 控制“趋势观察”。
- `showCommunities` 控制“主题社区”。
- `showPropagation` 控制“传播节点”。
- `showOrphanBridge` 同时控制“孤立文档 / 沉没文档 / 桥接节点”。

## 3. 各卡片统计规则、定义与使用场景

## 3.1 文档样本

### 统计规则

- 取 `filterDocumentsByTimeRange()` 的输出数量。
- 既包含当前窗口内有连接的文档，也包含当前窗口内新建/最近更新但尚未形成连接的文档。

### 定义

“文档样本”不是全库文档数，而是“本次分析真正参与计算的文档集合”。

### 详情列表

- 展示当前样本文档。
- 排序：最近更新时间倒序，其次标题升序。

### 使用场景

- 判断当前筛选器是否过窄或过宽。
- 解释为什么某些文档会出现在孤立文档卡片里，即使它们当前没有引用关系。

## 3.2 活跃关系

### 统计规则

- 值来自 `summary.totalReferences`。
- 实现是 `countUniqueDirectedEdges(references)`。
- 同一文档对之间多次块级引用，只算 1 条有向关系。

例子：

- 若 `A` 的三个块都引用 `B`，只记为 1 条 `A -> B`。
- 若同时存在 `A -> B` 和 `B -> A`，则计为 2 条活跃关系。

### 定义

“活跃关系”表示当前窗口中真正存在的文档级连接规模，不等于原始 `refs` 行数。

### 详情列表

- 只列出当前窗口内参与过连接的文档。
- 每个文档显示：
  - 入链：当前窗口内有多少个不同文档指向它。
  - 出链：当前窗口内它指向了多少个不同文档。
  - 徽标：入链与出链之和，即“参与次数”。

### 使用场景

- 快速判断当前知识网络是否活跃。
- 对比“文档样本”和“活跃关系”，识别样本中有多少文档其实仍未进入网络。

## 3.3 核心文档

### 统计规则

- 先为每个文档统计当前窗口内的有效入链。
- 只保留 `inboundReferences > 0` 的文档进入 `ranking`。
- 排序规则：
  1. `inboundReferences` 倒序
  2. `distinctSourceDocuments` 倒序
  3. `lastActiveAt` 倒序

实现细节：

- 当前实现里 `inboundReferences` 与 `distinctSourceDocuments` 实际是同一个值，都是“不同来源文档数”。
- 也就是说，核心文档排名按“有多少个不同文档引用它”来排，而不是按块级总引用次数排。

### 定义

核心文档 = 当前窗口内被至少一个其他文档引用的文档。

更准确地说：

- 它代表“被文档网络关注的中心文档”。
- 不是“编辑最多”或“出链最多”的文档。

### 详情列表

- 直接使用 `report.ranking`。
- 若文档本身已被识别为主题文档，则不再显示“升级为主题页”建议。
- 非主题文档在满足阈值时可能得到建议：
  - 条件：`inboundReferences >= 2` 或 `distinctSourceDocuments >= 2`
  - 仅取前 5 个建议项

### 使用场景

- 找索引页、总览页、主题页候选。
- 找当前窗口中最有“被引用价值”的知识入口。
- 结合引用证据面板查看“谁在引用它、最近何时引用”。

## 3.4 趋势观察

### 统计规则

- 在 `useAnalyticsState()` 中，趋势比较窗口 `trendDays` 取值为：
  - 当前 `timeRange` 为 `all` 时，固定 30 天；
  - 否则直接取当前时间窗口天数。
- `analyzeTrends()` 会比较：
  - 当前窗口
  - 前一个等长窗口

文档趋势口径：

- `currentReferences` / `previousReferences` 按“不同来源文档数”统计每个目标文档被引用情况。
- `delta = currentReferences - previousReferences`
- `delta > 0` 进入上升文档。
- `delta < 0` 进入回落文档。

连接变化口径：

- 按无向文档对比较新出现和消失的连接。

社区趋势口径：

- 先基于当前样本文档的“全量历史引用”建社区。
- 再分别统计这些社区在当前窗口和前一窗口中的内部唯一有向边数量。
- `delta > 0` 为上升社区。
- `currentReferences === 0`，或 `delta < 0 且 currentReferences <= 1`，归入沉寂社区。

### 定义

趋势观察反映的不是“文档修改趋势”，而是“文档级连接活跃度变化”。

### 详情内容

- 当前窗口活跃关系数
- 前一窗口活跃关系数
- 新增连接数
- 断裂连接数
- 上升文档、回落文档
- 上升社区、沉寂社区
- 断裂连接明细

### 使用场景

- 判断某个主题最近是否开始升温。
- 识别近期失活的关系或社区。
- 做阶段复盘，比如“近 7 天与前 7 天相比，哪些知识节点更常被提到”。

## 3.5 主题社区

### 统计规则

社区构建过程：

1. 用当前窗口有效引用建立无向邻接图。
2. 先识别桥接节点。
3. 在构建社区时，把桥接节点从遍历中排除。
4. 对剩余图做 BFS，得到连通分量。
5. 只保留文档数大于 1 的分量作为社区。

社区附加属性：

- `size`：社区文档数。
- `hubDocumentIds`：社区内按连接度排序取前 3 个文档。
- `topTags`：社区内出现频率最高的前 3 个标签。
- `notebookIds`：社区涉及到的笔记本集合。
- `missingTopicPage`：社区中是否缺少“主题页候选”。

主题页候选识别规则：

- 文档标题或标签包含以下关键词之一：
  - `index`
  - `hub`
  - `map`
  - `overview`
  - `guide`
  - `索引`
  - `导航`
  - `地图`
  - `总览`

### 定义

主题社区 = 去掉桥接节点后仍然内部紧密连接的文档簇。

它更接近“一个相对纯净的主题块”，而不是整个图的原始连通分量。

### 详情列表

- 按社区展开到文档粒度显示。
- 每个文档会带上：
  - 社区标签
  - 社区规模
  - 是否“缺主题页”标记

### 使用场景

- 识别稳定主题簇。
- 判断某一组笔记是否已经形成可独立维护的专题。
- 发现“社区已经形成，但还没有主题页/索引页”的结构缺口。

## 3.6 孤立文档

### 统计规则

- 基于当前窗口有效引用构建邻接图后，取 `degree === 0` 的文档。
- 这些文档来自当前样本，因此“最近新建/更新但没有连接的文档”也会进入孤立文档。

孤立文档附加字段：

- `historicalReferenceCount`：在当前筛选样本内，历史上与多少个不同文档发生过连接。
- `lastHistoricalAt`：最近一次历史连接时间。
- `hasSparseEvidence`：历史连接数在 `1~2` 之间。

排序支持：

- 最近更新倒序 `updated-desc`
- 创建时间倒序 `created-desc`
- 标题升序 `title-asc`

### 定义

孤立文档 = 当前窗口内没有任何有效文档级入链或出链的文档。

关键边界：

- 只看当前窗口，不看更早历史是否曾经连通过。
- 历史上曾经有连接，只要当前窗口没有有效连接，仍然会回到孤立文档。
- 同文档自引用不算连接。

### 详情列表

- 展示最近更新时间、创建时间。
- 根据建议系统显示“补齐链接”建议。
- 若命中主题文档匹配，还会在孤立详情中提供主题链接修复建议。

建议文案逻辑：

- 若 `hasSparseEvidence = true`：
  - “当前窗口内没有文档级连接，但历史上还有 X 条零散引用证据”
- 否则：
  - “当前分析窗口内没有文档级连接”

### 使用场景

- 找到刚写完但还没纳入体系的笔记。
- 找到曾经参与过网络、最近又掉出网络的文档。
- 用主题建议或手工补链，把文档重新接回主题网络。

## 3.7 沉没文档

### 统计规则

- 沉没文档是孤立文档的子集。
- 先取全部孤立文档，再计算未活跃天数：
  - `lastRelevantAt = max(updatedAt/createdAt, lastHistoricalAt)`
  - `inactivityDays = now - lastRelevantAt`
- 满足 `inactivityDays >= dormantDays` 即进入沉没文档。
- 默认阈值 `dormantDays = 30`，可在界面调整。

### 定义

沉没文档 = 当前窗口内无连接，且已连续达到阈值天数未产生有效连接的文档。

实现边界要特别说明：

- 代码没有强制要求“必须存在历史连接”。
- 因此，一个长期未更新、也从未连入网络的老文档，也可能进入沉没文档。
- 换句话说，当前实现更接近“长期失活的孤立文档”。

### 详情列表

- 展示未活跃天数。
- 若有稀疏历史证据，会显示“X 条历史连接”徽标。
- 显示“归档沉没”建议。

### 使用场景

- 做归档清理。
- 找长期未维护且未被网络引用的旧文档。
- 区分“只是暂时没接入网络的近期文档”和“已经长期沉底的旧文档”。

## 3.8 桥接节点

### 统计规则

桥接识别分两步：

1. 先找无向图中的 articulation points（割点）。
2. 再过滤成“有业务意义的桥接节点”：
   - 节点连接度至少为 3；
   - 移除该节点后，剩余图中至少出现 2 个规模 >= 2 的连通分量。

也就是说，当前实现不会把“只连着两个小尾巴”的轻微割点直接算作桥接节点。

### 定义

桥接节点 = 断开后会显著削弱多个文档簇之间连通性的关键文档。

### 详情列表

- 展示连接度 `degree`。
- 显示“重点维护”建议：
  - “连接 X 条关系，移除后会打断社区连通性”

### 使用场景

- 找需要重点维护的目录页、过渡页、索引页。
- 识别单点脆弱结构，避免某篇关键文档失效后导致主题网络断裂。
- 为“主题社区”拆分与解释提供依据。

## 3.9 传播节点

### 统计规则

传播节点不是标准 betweenness centrality，而是当前产品中的可解释启发式实现。

焦点文档集合 `focusIds` 由三部分组成：

- 核心文档排行前 12 个文档。
- 全部桥接节点。
- 每个社区的 hub 文档。

随后对每一对焦点文档：

1. 计算最短路径距离。
2. 只考虑距离 `>= 2` 的焦点对。
3. 对所有候选文档，若满足：
   - 不是这对焦点中的端点；
   - 同时位于这对焦点的某条最短路径上；
   - 即 `dist(left, candidate) + dist(candidate, right) === dist(left, right)`；
4. 则该候选文档记 1 次传播命中。

最终生成字段：

- `pathPairCount`：命中的焦点对数量。
- `score`：当前实现里等于 `pathPairCount`。
- `focusDocumentCount`：它覆盖了多少个不同焦点文档。
- `communitySpan`：它跨到了多少个社区。
- `bridgeRole`：它本身是否也是桥接节点。
- `degree`：当前窗口无向连接度。

排序规则：

1. `score` 倒序
2. `communitySpan` 倒序
3. `degree` 倒序
4. `documentId` 升序

### 定义

传播节点 = 高频出现在关键焦点文档之间最短路径上的中间文档。

它表达的是：

- 哪些文档最容易成为“关系传播经过点”
- 哪些文档最适合补充上下游导航、路径说明、串联解释

### 详情列表

- 徽标显示 `score` 分。
- 元信息显示：
  - 覆盖多少个焦点文档
  - 社区跨度
- 固定附带“传播优化”建议。
- 若 `bridgeRole = true`，建议文案会额外强调其桥接角色。

### 使用场景

- 找适合作为“路径解释页”的文档。
- 补强跨主题导航。
- 识别对关系传播效率最敏感的中间层文档。

## 4. 建议系统与卡片的关系

建议生成逻辑在 `buildSuggestions()`。

当前建议类型与触发来源：

| 建议类型 | 来源卡片 | 触发条件 |
| --- | --- | --- |
| `promote-hub` 升级为主题页 | 核心文档 | 满足核心文档建议阈值，最多前 5 个 |
| `repair-orphan` 补齐链接 | 孤立文档 | 所有孤立文档 |
| `archive-dormant` 归档沉没 | 沉没文档 | 所有沉没文档 |
| `maintain-bridge` 重点维护 | 桥接节点 | 所有桥接节点 |

传播节点的建议不走 `report.suggestions`，而是在详情构建阶段动态生成固定文案。

## 5. 卡片点击后的详情联动规则

联动链路：

- 顶部卡片点击后修改 `selectedSummaryCardKey`
- `selectedSummaryDetail` 从 `summaryDetailSections` 中取对应 section
- `src/App.vue` 根据 `kind` 渲染不同详情面板

特殊交互：

- 孤立文档卡片：
  - 支持切换排序方式。
  - 支持主题建议链接的插入与撤销。
- 沉没文档卡片：
  - 支持调整 `dormantDays`，会立即影响统计结果。
- 趋势卡片：
  - 展示上升/回落文档、社区变化、连接断裂明细。
- 传播节点卡片：
  - 单独使用传播详情样式，突出分数与传播建议。

## 6. 几个容易误解但当前代码就是这样实现的点

1. “活跃关系”不是引用行数，而是唯一有向文档边数量。
2. “核心文档”的入链统计按不同来源文档去重，不按块级引用次数累计。
3. “孤立文档”只看当前窗口，不看历史是否曾经连通过。
4. “沉没文档”当前实现并不要求历史上一定有连接，它只是达到未活跃阈值的孤立文档。
5. “主题社区”是先剔除桥接节点后得到的主题簇，因此它有意弱化了跨社区连接。
6. “传播节点”是产品启发式评分，不是严格图论中心性。
