# 孤立文档编辑后消失的真凶：一次被 UTC 坑到的排障复盘

前两天我碰到一个特别像“产品语义写错了”的 bug。

用户反馈说，某篇孤立文档先通过“补齐主题链接”变成了非孤立文档，这一步完全正常。接着又把链接删掉，再点“刷新分析”，结果这篇文档并没有回到“孤立文档”里。

如果事情只到这里，其实还不算离谱。更离谱的是，再继续看界面，会发现它连“文档样本”里都不在了。

这就很微妙。

因为从直觉上说，一篇文档删掉连接之后，最差的结果也应该是“被错分到别的类别”，而不是整篇文档像蒸发了一样，从分析结果里直接消失。

这种 bug 最容易让人上来就改业务逻辑。比如怀疑“孤立文档”和“沉没文档”的定义是不是混了，或者怀疑删除链接之后数据库里还有残留引用没有清干净。

但这次真正的教训恰恰是，越像业务 bug 的问题，越要先把数据层查穿，再动代码。

## 一开始，所有线索都在把人往错误方向带

我先说为什么这个问题特别像“孤立判定逻辑有问题”。

在当前插件里，用户看到的是“孤立文档”这个业务标签。一个文档加了链接，不再孤立。删掉链接，理应重新变回孤立。这个预期太自然了，所以只要结果不对，大家第一反应都会认为是孤立判定代码出错。

再加上这个功能链路里本来就有几层容易混淆的概念：

1. 当前窗口内没有连接。
2. 历史上从未有过连接。
3. 当前不活跃，但历史上可能连接过。

只要这三层里有任何一层没处理好，界面就会表现得很奇怪。所以一开始往“分类逻辑出错”去想，完全合理。

但我很快注意到一个不对劲的地方。用户补充了一句：刷新后，这篇文档不只是没出现在“孤立文档”里，连“文档样本”里也没有。

这句话其实非常关键。

因为一旦一个文档连样本都进不去，那就说明问题已经不在“它属于哪一类”，而在“它为什么没有进入分析集合”。

换句话说，这不是分类问题，而是样本问题。

很多排障就是这样。真正有价值的突破口，经常不是一大堆日志，而是用户随口补充的一句现象描述。只要你把这句话听进去，整个排查方向就会变。

## 先别猜，先用 MCP 把数据库状态查实

既然问题可能不在分类层，那第一步就不能再靠界面猜了。我直接用 MCP 去查思源里的真实数据。

目标文档是 `20260221212452-gwpyddn`。

我做了几件事。

先读文档内容和子块，确认正文里到底还有没有 `siyuan://blocks/...` 或 `((...))` 这类内部链接。结果没有。

再查 `refs` 表，确认它是否还保留着入链或出链。结果也没有。

最后查反链，看看有没有其它文档还在指向它。结果仍然是空。

这一步做完以后，事情就清楚了一半。因为数据库层已经可以确认，这篇文档当前就是一个标准的“无连接文档”。

注意这个结论有多重要。

它直接排除了两个最容易让人纠缠很久的猜测：

1. 不是“链接看起来删了，其实 refs 里还有残留记录”。
2. 不是“文档其实还有别的文档指向它，所以本来就不该回到孤立文档”。

到这里，问题就从“为什么它不是孤立文档”，收缩成了一个更精确的问题：

既然它在数据库里已经没有连接，那为什么刷新分析后，它会从文档样本里消失。

这一步一旦想明白，排障就会快很多。因为样本问题和分类问题，对应的是完全不同的代码路径。

## 真正的问题，不在孤立判定，而在时间窗口筛样本

顺着这条线往下看，前端里的“文档样本”来自 `filteredDocuments`，它的核心逻辑在 `filterDocumentsByTimeRange()`。

这段逻辑本身并不复杂。一个文档要进入当前分析样本，只要满足下面两个条件之一：

1. 当前时间窗口内有有效引用关系。
2. 文档自己的 `updated` 或 `created` 时间落在当前时间窗口里。

目标文档在删除链接以后，已经没有任何连接了，所以它想进入样本，只能靠第二条。

而偏偏它又是“刚编辑过”的文档。按理说，越是刚编辑过，越应该稳稳落进 `7d` 或 `3d` 的时间窗口里。

结果它却被排除了。

这时候就有意思了。因为这说明不是窗口规则太严，而是“时间本身被解释错了”。

继续往里看，果然在时间戳解析函数里发现了问题。原来的实现用的是 `Date.UTC(...)`。

这行代码单独看并不显眼，甚至还挺像“标准写法”。但放在思源的场景里，它恰恰是错的。

思源给出来的时间戳，像 `20260313213430`，表达的是本地时间。也就是北京时间 `2026-03-13 21:34:30`。

可一旦你用 `Date.UTC()` 去解析，它就不再表示“北京时间晚上九点半”，而会被当成“UTC 时间晚上九点半”。换算到本地时区，相当于第二天凌晨五点半。

问题来了。

如果当前系统时间是 `2026-03-13 22:02:45 +08:00`，那么这个被错误解析后的时间，不是“半小时前编辑过”，而是“未来八小时之后才会发生”。

而时间窗口判断又写成了“必须小于等于当前时间”，于是这个文档会被直接排除出样本。

这就是整个 bug 最反直觉的地方。

用户看到的是“我编辑了一篇孤立文档，刷新后它没了”。真正发生的却是“因为你刚刚编辑过，它的更新时间被错误解析成了未来时间，于是样本筛选把它踢掉了”。

这类 bug 最容易折腾人的地方就在这里：现象和根因完全不在一个层级上。

## 为什么这个 bug 很像业务问题，却偏偏藏在基础函数里

回头看这个 bug，它其实具备几个非常典型的“误导性特征”。

第一，它不是普遍出错，而是条件触发。

不是所有孤立文档都会消失，只有“当前没有连接，同时又刚刚被编辑过”的文档才会触发。只要文档当前还有连接，它就可以通过“活跃引用”这条路径进入样本，时间解析错误就被遮住了。

第二，它发生在一个业务概念特别强的界面里。

用户看到的是“孤立文档”，开发者自然就会先盯着“孤立文档判定”。但真正的 bug 发生在更底层的样本筛选阶段。高层名词越明确，越容易把人锁死在错误层级里。

第三，它的数据层表现是干净的。

这类情况最容易让人怀疑“前端状态缓存没刷新”，或者“界面展示有残留状态”。但这次恰恰不是缓存展示问题，而是新鲜的数据被错误地解释了。

所以这次排障给我最大的提醒是，遇到“某个分类不对”的问题时，先问自己一句：这个对象到底还在不在分析集合里。

只要这一步没有确认，就不要急着改分类逻辑。

## 修复只改了一行，但真正重要的是定位方式

最后的修复其实非常小。

把时间解析从 UTC 改成本地时间：

```ts
return new Date(year, month, day, hour, minute, second).getTime()
```

改完以后，刚编辑过的无连接文档不会再被错误判成“未来时间”，于是它能正常进入时间窗口样本，也会正常回到孤立文档统计。

但说实话，这次真正有价值的，不是“把这一行改对了”，而是整个定位过程。

如果一开始我顺着界面现象去改“孤立文档定义”，或者去补一堆“删除链接后重新归类”的业务判断，大概率只会把问题越改越绕。

真正起作用的是下面这几个动作：

先用 MCP 把文档内容、`refs`、反链查实。

再确认它到底是“分类错误”，还是“样本缺失”。

最后才去找那个决定样本进入条件的最底层函数。

很多时候，一个 bug 难，不是因为它修起来难，而是因为它特别会伪装。

它把自己伪装成业务语义问题，伪装成缓存问题，伪装成 refs 清理不完整的问题。你只要跟着其中任何一个假象走太远，就会在错误方向上投入很多时间。

所以这次我最想记住的，不是 UTC 这个具体坑，而是一条更通用的经验：

当一个对象“看起来被分错了”，先别急着研究它为什么分错。先确认它到底有没有资格被分。

这句话听起来有点绕，但在实际排障里非常有用。因为很多分类问题，最后根本不是分类错，而是样本先没了。
